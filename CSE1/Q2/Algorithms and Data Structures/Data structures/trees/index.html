Some more terminology:

  • Depth of a node: the number of ancestors of that node, other than the node itself. For H, this would be 3.
  • Height of a tree: The maximum of the depths of its nodes. So in this case, 3.


Traversals

Tree traversal: a systematic way of visiting all nodes of a tree


There are a few kinds of tree traversals:


Depth first traversal:

Pre-order traversal:

  1. Perform the "visit" action (can be whatever)
  2. For each child of the current node, perform pre-order traversal on that child (recursively traverse the subtree rooted at that child)


Post-order traversal:

  1. For each child of the current node, perform post-order traversal on that child
  2. Perform the "visit" action


In-order traversal (only for binary trees, see the next part):

  1. If the current node has a left child then perform in-order traversal on that child
  2. Perform the "visit" action
  3. If the current node has a right child then perform in-order traversal on that child


Breadth-first traversal: Starts at the root, and then goes 'down' the tree by visiting all neighbor nodes from left to right and then going one level down.


Binary trees

A binary tree is a tree which has the following properties:

  • Every node has at most two children
  • Every child node is labeled left child or right child
  • A left child precedes a right child in the order of children of a node

A proper (full) binary tree is a tree in which each node either has 0 or two children

A complete binary tree is a tree where each level, except possibly the last if completely filled and all nodes in the last level are as far left as possible.

A balanced tree is a tree where the left and right subtrees' heights differ by at most one

If a node doesn't have a child, the reference to that child will just be null, so beware of that.


Implementations

Linked binary tree: A logical way of realizing a binary tree is by using a linked structure, where each node points to a left, right and a parent node. It also contains a reference to its own value (its element). The root would have a null reference to its parent. Many operations on this implementation could be done in ﻿O(1)O\left(1\right)O(1)﻿ time (just the height takes ﻿O(n)O\left(n\right)O(n)﻿). It takes ﻿O(n)O\left(n\right)O(n)﻿ space.


To use this linked implementation of a general tree (which can have more than 2 children), we could use an array of ArrayList for keeping track of the children (instead of a "left" and "right" reference)


Array-based binary tree: Another way of implementing a binary tree is using an array, where a node is stored at a logical index.

Binary search trees

A binary search tree is a binary tree in which each node with key ﻿kkk﻿ satisfies the following properties:

  • Keys stored in the left subtree are less than ﻿kkk﻿
  • Keys stored in the right subtree are greater than ﻿kkk﻿


Make sure that this if also true for all the children of this tree! It is not a binary search tree anymore if for example the root is 30, its left child is 15 (so far so good), but then the right child of that left child is 31.


In this way binary search is pretty easy: if you search a smaller value, go left etc.


Inserting into binary search tree: you must find the position for inserting. If it already exists in the tree, we must just update its value. Else, we insert a new node by expanding the leaf that was reached after failing to search for the node.


Removing from binary search tree:

  • Easy case: at most one of the children of the to be removed node is internal, then we can remove the node and replace it by the child.
  • Difficult case: both children of the to be removed node are internal
  • Find the maximal node in the left child (the greatest key less than the to key of the to be removed node)
  • Replace the to be removed node by the maximal node (so the search-tree property is still satisfied)
  • Recursively proceed to remove that maximal node (so repeat this until the to be removed node has a maximum of one child)


So the worst time complexity of searching for a key in a balanced tree is ﻿O(log⁡2n)O\left(\log_2n\right)O(log2​n)﻿ and in a non-balanced tree ﻿O(n)O\left(n\right)O(n)﻿


AVL trees

Condition: the heights of the children of each node differ by at most 1.


An AVL tree also must be a search tree.


We consider null references as leaves when counting heights.


Inserting

  • Use ordinary binary tree insertion
  • This could create unbalanced nodes, but only along the path to the inserted node
  • Then execute search and repair while traversing to the root
  • Then fix unbalanced nodes using tri-node restructuring along the way (see the picture), while traversing to the root, for each node there are three cases:
  • ﻿∣h1−h2∣≤1\left|h_1-h_2\right|\le1∣h1​−h2​∣≤1﻿: AVL property holds
  • ﻿∣h1−h2∣=2\left|h_1-h_2\right|=2∣h1​−h2​∣=2﻿: found node to rebalance
  • ﻿∣h1−h2∣≥3\left|h_1-h_2\right|\ge3∣h1​−h2​∣≥3﻿: impossible, insertion can change the height by at most one

A double rotation is called this way as it can be achieved by performing two rotations, where a single rotation can be achieved by only one rotation.


Removal

  • Use ordinary binary tree insertion
  • This could create unbalanced nodes, but only along the path to the removed node
  • Then execute search and repair while traversing to the root
  • Then fix unbalanced nodes using tri-node restructuring along the way

get, put, remove are O(logn)\mathcal{O}(logn)O(logn)

  • The height of an AVL tree of size ﻿nnn﻿ is ﻿O(log⁡n)O\left(\log n\right)O(logn)﻿ (for proof see the slides / book)
  • Restructuring takes ﻿O(1)O\left(1\right)O(1)﻿ time
  • Getting, putting or removing visits ﻿O(h)O\left(h\right)O(h)﻿ nodes and performs ﻿O(h)O\left(h\right)O(h)﻿ restructurings


(2, 4) trees

Multi-way search trees: ordered trees such that an internal node is a ﻿ddd﻿-node:

  • it has ﻿ddd﻿ children
  • stores an ordered set of ﻿d−1d-1d−1﻿ items
  • for each key ﻿kik_iki​﻿, all keys in the subtree rooted at child ﻿cic_ici​﻿ are smaller than ﻿kik_iki​﻿, and all keys in the subtree rooted at child ﻿ci+1c_{i+1}ci+1​﻿ are larger than ﻿kik_iki​﻿

Leaves store no items


A (2,4) tree is a multi-way search tree with 2 additional properties:

  • Size property: every internal node has at most 4 children, this keeps nodes simple, with only a few items and children, resulting in ﻿O(1)O\left(1\right)O(1)﻿ time node operations
  • Depth property: all external nodes have the same depth, this ensures that the height of a (2,4) tree storing ﻿nnn﻿ entries is ﻿O(log⁡n)O\left(\log n\right)O(logn)﻿



Insertion

  • Insert new item at the parent ﻿www﻿ of the leaf reached by searching for key (this may cause an overflow, as the parent might become a 5-node)
  • Repair an overflow by a split:
  • Replace ﻿www﻿ by two nodes ﻿w′w&#x27;w′﻿ (3-node with ﻿c1, c2, c3c_1,\ c_2,\ c_3c1​, c2​, c3​﻿ and keys ﻿k1, k2k_1,\ k_2k1​, k2​﻿), and ﻿w′′w&#x27;&#x27;w′′﻿ (2-node with ﻿c4, c5c_4,\ c_5c4​, c5​﻿ and ﻿k4k_4k4​﻿)
  • If ﻿www﻿ is the root of the tree, create a new node ﻿uuu﻿, else let ﻿uuu﻿ be the parent of ﻿www﻿
  • Insert key ﻿k3k_3k3​﻿ into ﻿uuu﻿ and make ﻿w′w&#x27;w′﻿ and ﻿w′′w&#x27;&#x27;w′′﻿ children ﻿iii﻿ and ﻿i+1i+1i+1﻿ of ﻿uuu﻿ respectively
  • An overflow can propagate to the parent

Deletion

  • Removing the ﻿ithi^{th}ith﻿ item ﻿(ki, vi)\left(k_i,\ v_i\right)(ki​, vi​)﻿ of node ﻿zzz﻿ with internal children is reduced to removing an item at node ﻿www﻿ with only external children
  • Find the rightmost internal node ﻿www﻿ in the left subtree of the ﻿ithi^{th}ith﻿ item (rooted at the ﻿ithi^{th}ith﻿ child of ﻿zzz﻿)
  • Swap the item ﻿(ki, vi)\left(k_i,\ v_i\right)(ki​, vi​)﻿ at node ﻿zzz﻿ with the last item of node ﻿www﻿
  • Remove item ﻿(ki, vi)\left(k_i,\ v_i\right)(ki​, vi​)﻿ from node ﻿www﻿ and remove the external that is the ﻿ithi^{th}ith﻿ child of ﻿www﻿
  • Deletion deserves the depth property, but can break the size property, causing an underflow (as in the parent there are still 2 keys, but it now has only 2 childs)


Fixing this underflow can be put in 2 cases:

  • Adjacent siblings of node ﻿www﻿ are 2-nodes
  • Repair by fusion
  • Merge ﻿www﻿ with a sibling
  • Move a key from the parent ﻿uuu﻿ of ﻿www﻿ to the merged node-
  • Can propagate underflow to parent
  • Adjacent sibling ﻿sss﻿ of node ﻿www﻿ is a 3- or 4-node
  • Repair by transfer
  • Move a child of ﻿sss﻿ to ﻿www﻿
  • Move a key of ﻿sss﻿ to the parent ﻿uuu﻿ of ﻿sss﻿ and ﻿www﻿
  • Move a key of ﻿uuu﻿ to ﻿www﻿
  • After repair, no underflow still occurs


Performance

Similar to AVL tree, time depends on the height (﻿O(log⁡n)O\left(\log n\right)O(logn)﻿)


Red-black trees

Definition: A red-black tree is a binary search tree that satisfies the following properties (Arie's definition: a red-black tree is a nice sorted tree with properties):

  • Root property: the root is black
  • External property: every leaf is black
  • Internal property: the children of a red node are black
  • Depth property: all leaves have the same black depth (number of proper ancestors that are black)


Converting into (2, 4) tree

  • A black node has at most 2 red children, a black node and its red children can be merged into a (2, 4) node
  • If a black node has no red children, it will become a 2-node. With 1 red child it will become a 3-node
  • Since all external nodes have the same black depth, the resulting (2, 4) tree is well formed


Insertion

To insert an item: do binary search, insert new node ﻿xxx﻿. Color ﻿xxx﻿ black if it is the root, color ﻿xxx﻿ red otherwise.

  • Preserves the root, external, and depth properties.
  • If ﻿xxx﻿ is not the root (thus ﻿xxx﻿ is red) and its parent ﻿yyy﻿ is black, internal property is preserved and we are done.
  • Otherwise (both ﻿xxx﻿ and parent ﻿yyy﻿ are red) we have a double red, which requires a reorganization of the tree.


This reorganization can be put in 2 cases:

  • The sibling ﻿sss﻿ of parent ﻿yyy﻿ is black: repair by trinode restructuring
  • The sibling ﻿sss﻿ of parent ﻿yyy﻿ is red
  • Repair by recoloring: color ﻿yyy﻿ and ﻿sss﻿ black, parent ﻿zzz﻿ of ﻿yyy﻿ red
  • Double red problem may propagate to ﻿zzz﻿

Deletion

To remove an item, we first execute the deletion algorithm for binary search trees.

This deletes an internal node ﻿vvv﻿ and an external node ﻿www﻿, and promotes sibling ﻿ppp﻿ of ﻿www﻿.

  • If either ﻿vvv﻿ or ﻿ppp﻿ was red, color ﻿ppp﻿ black and we are done.
  • Else (﻿vvv﻿ and ﻿ppp﻿ both black), color ﻿ppp﻿ double black to preserve depth property. However, there’s an underflow in the equivalent (2,4) tree, which requires a reorganization of the tree.


This repair of a double black node ﻿ppp﻿ with sibling ﻿yyy﻿ is split up into 3 cases

  • ﻿yyy﻿ is black and has a red child ﻿xxx﻿: perform trinode restructuring, equivalent to a transfer in the associated (2, 4) tree, and we're done
  • ﻿yyy﻿ is black and its children are both black:
  • Change the color of ﻿ppp﻿ from double black to black, and the color of ﻿yyy﻿ from black to red. Preserves red property
  • To counteract the decrease in black depth for paths passing through ﻿yyy﻿ or ﻿ppp﻿, we consider their common parent ﻿zzz﻿:
  • If ﻿zzz﻿ is red, color it black. The problem is resolved.
  • If ﻿zzz﻿ is black, color it double black. The problem is propagated higher up.
  • ﻿yyy﻿ is red: perform an adjustment
  • Let ﻿zzz﻿ be the common parent of ﻿yyy﻿ and ﻿ppp﻿ (note that ﻿zzz﻿ must be black, since ﻿yyy﻿ is red).
  • The combination of ﻿yyy﻿ and ﻿ppp﻿ represents a 3-node in the corresponding (2,4) tree, which needs to be reoriented.
  • Rotate ﻿yyy﻿ to be above its parent ﻿zzz﻿. Recolor ﻿yyy﻿ black and ﻿zzz﻿ red.
  • After the adjustment, either Case 1 or Case 2 applies.

Performance

Similar to AVL and (2,4) tree


AVL vs (2,4) vs Red-black trees

  • Insertion and removal is faster with red-black trees than AVL and (2,4) trees, since an insertion or deletion in a red-black tree requires only a constant number of structural changes (restructurings), plus an ﻿O(log⁡n)O\left(\log n\right)O(logn)﻿ number of recolorings.
  • Searching is faster in AVL and (2,4) trees, since they are better balanced.
