Prefix: sorting with a priority queue

The next three sorting algorithms (selection sort, insertion sort, heap sort) all use the following two-phase way of sorting.


An application of a priority queue is sorting, which can be done in the following way:

  1. Insertion phase: Insert the elements of a sequence ﻿SSS﻿ as keys into an initially empty priority queue ﻿PPP﻿ by insert operations, one for each element
  2. Removal phase: Call removeMin on ﻿PPP﻿ and put that element back in ﻿SSS﻿


Selection sort

Time complexity:

Sorting elements using an unsorted list priority queue:

  1. Insertion into ﻿PPP﻿ is ﻿O(1)O\left(1\right)O(1)﻿, so ﻿nnn﻿ insertions gives ﻿O(n)O\left(n\right)O(n)﻿
  2. The removal phase runtime performance is proportional to the size of ﻿PPP﻿, giving a total of ﻿O(n2)O\left(n^2\right)O(n2)﻿


Space complexity:

The space complexity depends on the implementation. If the to be sorted sequence ﻿SSS﻿ is a linked list whose size is modified by insert/remove, the total number of elements in ﻿SSS﻿ and ﻿PPP﻿ combined is always equal to the original number in ﻿SSS﻿, giving a space complexity of ﻿O(1)O\left(1\right)O(1)﻿.

If the to be sorted sequence is an array list of capacity ﻿nnn﻿ (or a linked list that doesn't not shrink, elements are overridden), we need ﻿O(n)O\left(n\right)O(n)﻿ space.


In-place:

Selection sort can be implemented in-place:

  • Iterate left to right, one index ﻿iii﻿ at a time (where ﻿i = 0; i &lt; n−1i\ =\ 0;\ i\ &lt;\ n-1i = 0; i < n−1﻿)
  • Find index ﻿mmm﻿ of minimum element within indexes ﻿i,...,n−1i,...,n-1i,...,n−1﻿
  • Swap elements at ﻿iii﻿ and ﻿mmm﻿


Insertion sort

Time complexity:

Sorting elements using a sorted list priority queue:

  1. Insertion runtime performance is proportional to the size of ﻿PPP﻿ , giving a total of ﻿O(n2)O\left(n^2\right)O(n2)﻿
  2. Removal from the queue is ﻿O(1)O\left(1\right)O(1)﻿, so ﻿nnn﻿ removals gives ﻿O(n)O\left(n\right)O(n)﻿


Space complexity:

The space complexity is the same as for selection sort.


In-place:

Insertion sort can also be implemented in-place, so the space complexity is always ﻿O(1)O\left(1\right)O(1)﻿. This can be done the following way:

  • Iterate left to right, one index ﻿iii﻿ at a time (where ﻿i =1; i &lt;ni\ =1;\ i\ &lt;ni =1; i <n﻿)
  • Iterate backwards from that ﻿iii﻿ to ﻿000﻿ and when the current element is larger than the element at index ﻿iii﻿, shift a position to the right. When the right position is found, insert the element at ﻿iii﻿ in that position


Heap sort

Time complexity:

Sorting elements using a min-heap-based priority queue:

  1. The ﻿iii﻿th insert operation takes ﻿O(log⁡2i)O\left(\log_2i\right)O(log2​i)﻿ time, since the heap has ﻿iii﻿ entries after the operation. Therefore, this phase takes ﻿O(nlog⁡2n)O\left(n\log_2n\right)O(nlog2​n)﻿ time (which could be ﻿O(n)O\left(n\right)O(n)﻿ using bottom-up heap construction). So after this, all elements are inserted into a min-heap.
  2. The removal from the heap can be done by repeated calls to removeMin. The ﻿jjj﻿th remove operation takes ﻿O(log⁡2(n−j+1))O\left(\log_2\left(n-j+1\right)\right)O(log2​(n−j+1))﻿ since the heap has ﻿n −j+1n\ -j+1n −j+1﻿ entries. This phase takes ﻿O(nlog⁡2n)O\left(n\log_2n\right)O(nlog2​n)﻿ (which is the complexity for the entire algorithm).


Space complexity:

The space complexity is ﻿O(n)O\left(n\right)O(n)﻿.


In-place:

In order to achieve in-place heap sorting we need to use a portion of the array for the heap and the remaining portion for the sorted sequence. This can be done as follows:

  1. We use a max-heap instead of a min-heap. At any time during the execution of the algorithm, we use the left portion of ﻿SSS﻿, up to a certain index ﻿i−1i-1i−1﻿ to store the entries of the heap, and the right portion to store the elements of the sequence. So the first ﻿iii﻿ elements provide the array-list representation of the heap
  2. In the first phase of the algorithm, we start with an empty heap and move the boundary between the heap and sequence to right, to we end up with the entire array used for the heap
  3. In the second phase of the algorithm, we start with an empty sequence and we move this boundary to the left again, one step at a time, by removing the maximal element from the heap and storing it at index ﻿n−in-in−i﻿.


Prefix: divide-and-conquer

The next two sorting algorithms (merge sort, quick sort) all use the following three-phase way of sorting.


Divide-and-conquer is an algorithmic design pattern consisting of 3 steps:

  1. Divide:
  2. Small input: base case - If input is smaller than a certain threshold, solve the problem directly
  3. Larger input: recurrence case - Divide the input into two or more disjoint subsets (no common elements!)
  4. Conquer: recursively solve the sub-problems associated with the subsets
  5. Combine: Take the solutions of the sub-problems and merge them into a solution to the larger problem


Merge sort

Implementation:

Merge sort uses the divide-and-conquer pattern to sort a sequence ﻿SSS﻿ with ﻿nnn﻿ elements:

  1. Divide:
  2. If ﻿SSS﻿ has 0 or 1 elements, return ﻿SSS﻿ (already sorted)
  3. If ﻿SSS﻿ has at least 2 elements, split elements of ﻿SSS﻿ into 2 sequences: ﻿S1S_1S1​﻿ and ﻿S2S_2S2​﻿. These sequences contain each about half of the elements: ﻿S1S_1S1​﻿ the first ﻿⌊n2⌋\lfloor\frac{n}{2}\rfloor⌊2n​⌋﻿, ﻿S2S_2S2​﻿ the remaining ﻿⌈n2⌉\lceil\frac{n}{2}\rceil⌈2n​⌉﻿.
  4. Conquer: Recursively sort these sequences
  5. Put elements back into ﻿SSS﻿ by merging the sorted sequences into a sorted sequence (this is done by repeatedly getting the minimum of the first values of both the sorted arrays)


This means you get a lot of recursive calls and merges of sorted sequences into bigger sorted sequences.


Time complexity:

You can also visualize this sorting algorithm using a recursion tree (which contains every recursive call in a tree shape, see page 38 of the slides). The height ﻿hhh﻿ of this tree is ﻿O(log⁡2n)O\left(\log_2n\right)O(log2​n)﻿ (as at every level the amount elements is split in 2 halves). The overall work done at the nodes of a level ﻿iii﻿ is ﻿O(n)O\left(n\right)O(n)﻿, as we have to make ﻿2i+12^{i+1}2i+1﻿ recursive calls and partition and merge ﻿2i2^i2i﻿ sequences of size ﻿n2i\frac{n}{2^i}2in​﻿.


This gives us a total time complexity of ﻿O(nlog⁡2n)O\left(n\log_2n\right)O(nlog2​n)﻿


Quick sort

Implementation:

Quick sort uses the divide-and-conquer pattern to sort a sequence ﻿SSS﻿ with ﻿nnn﻿ elements. In quick sort (in contrary to merge sort), most of the work is done before the recursive calls:

  1. Divide:
  2. If ﻿SSS﻿ has less than 2 elements, return
  3. If ﻿SSS﻿ has at least 2 elements, select a specific element from ﻿SSS﻿, called the pivot. Common practice is to choose the pivot as the last element in ﻿SSS﻿. Then remove all elements from ﻿SSS﻿ and split them into 3 sequences:
  4. Sequence ﻿LLL﻿: elements from ﻿SSS﻿ that are smaller than the pivot
  5. Sequence ﻿EEE﻿: elements from ﻿SSS﻿ that are equal to the pivot
  6. Sequence ﻿RRR﻿: elements from ﻿SSS﻿ that are larger than the pivot
  7. Conquer: Recursively sort ﻿LLL﻿ and ﻿RRR﻿
  8. Combine: Put elements back into ﻿SSS﻿: first all elements of ﻿LLL﻿, then of ﻿EEE﻿, then of ﻿RRR﻿


Time complexity:

Again we can use a recursion tree (page 27 of the slides), and we see that in the good case every partition has almost the same size. Then the running time is the same as in merge sort (﻿O(nlog⁡2n)O\left(n\log_2n\right)O(nlog2​n)﻿). The space is ﻿O(log⁡2n)O\left(\log_2n\right)O(log2​n)﻿


In the worst-case, each recursive call, all the other elements are either bigger or smaller than the pivot. Then the height of the tree becomes ﻿O(n)O\left(n\right)O(n)﻿, the work per level is ﻿n−1n-1n−1﻿, which gives a total of ﻿O(n2)O\left(n^2\right)O(n2)﻿ time complexity and ﻿O(n)O\left(n\right)O(n)﻿ space.


Randomized quick sort

Instead of always picking a predetermined element as pivot, we use a random pivot, so we don't get the worst-case performance for certain distributions of input. It becomes tricky to analyze time complexity as we don't know how good our random pick is. What we can say is the following: an element ﻿xxx﻿ is involved as a non-pivot in only one comparison per level until it either:

  • becomes the pivot
  • becomes the only element


The total time complexity with a randomized pivot becomes ﻿O(nlog⁡2n)O\left(n\log_2n\right)O(nlog2​n)﻿, read slide 41 for a better justification, the gist of it is the following:

  • We take the sum ﻿∑s∈SC(x)\sum_{s\in S}^{ }C\left(x\right)∑s∈S​C(x)﻿ where ﻿C(x)C\left(x\right)C(x)﻿ is the number of comparisons involving ﻿xxx﻿ as a non-pivot element
  • We consider a pick of pivot good if the amount of elements in the next recursive call are at most ﻿34\frac{3}{4}43​﻿ of the original sequence. Then the chance of having a good pick is at least ﻿12\frac{1}{2}21​﻿.
  • There are most ﻿log⁡43n\log_{\frac{4}{3}}nlog34​​n﻿ good pivot choices before ﻿xxx﻿ is isolated. The expected number of choices to get that amount of good choices is ﻿2log⁡43n2\log_{\frac{4}{3}}n2log34​​n﻿, giving us ﻿C(x)C\left(x\right)C(x)﻿ is ﻿O(log⁡2n)O\left(\log_2n\right)O(log2​n)﻿.
  • Taking the sum of those terms gives a total time complexity of ﻿O(nlog⁡2n)O\left(n\log_2n\right)O(nlog2​n)﻿


In-place quick sort

A sub-sequence of the input is represented implicitly by the range given by indexes ﻿aaa﻿ and ﻿bbb﻿. The following changes must be made:

  • Select that last item in the current sequence as pivot
  • The divide step scans the array simultaneously:
  • forward, using index ﻿lll﻿
  • backward, using index ﻿rrr﻿
  • if an item with index ﻿rrr﻿ is lower than the pivot, switch it with an item larger than the pivot in index ﻿lll﻿ (if there is none, so index ﻿lll﻿ has been increasing without an item being larger than the pivot, then ﻿rrr﻿ was in the right place). For a great explanation on this, see here
  • When these indexes cross, division is complete and the algorithm recurs on the two sub-sequences. No explicit combine step is needed, we now also don't create 3 sequences, values equal to the pivot can be dispersed across the sub-lists.


Prefix: key-based sorting

The best we can do with sorting based on comparison is ﻿Ω(nlog⁡n)\Omega\left(n\log n\right)Ω(nlogn)﻿. There is a big explanation on this in the book at 13.3.1, which can be summarized by this:

  • Every comparison can only have one of 2 outcomes, "yes" or "no". Based on this, we can create a decision tree.
  • Each permutation of ﻿SSS﻿ will cause a series of comparisons, each external node in the tree can represent comparisons for at most one permutation of ﻿SSS﻿
  • The running time of this algorithm must be greater than or equal to the height of the decision tree. The number of permutations of ﻿nnn﻿ objects is ﻿n!n!n!﻿.
  • We must have at least ﻿n!n!n!﻿ external nodes, thus the height is at least ﻿log⁡2(n!)\log_2\left(n!\right)log2​(n!)﻿, giving us ﻿Ω(nlog⁡n)\Omega\left(n\log n\right)Ω(nlogn)﻿


But can we do better? Yes, in certain conditions we can, using a sequence of entries which are a key-value pair where the keys are of a restricted type.


Bucket sort

Implementation:

Bucket sort uses a sequence of ﻿nnn﻿ entries whose keys are integers in the range ﻿[0,N−1]\left[0,N-1\right][0,N−1]﻿, where ﻿N≥2N\ge2N≥2﻿ and the sequence should be sorted by those keys. Now we can sort in ﻿O(n+N)O\left(n+N\right)O(n+N)﻿ time.


This algorithm uses keys as indices into a bucket array, that has cells indexed from ﻿000﻿ to ﻿N−1N-1N−1﻿. Every index in that array is a sequence with all the elements with that specific key.

After inserting everything into that array, everything is removed from it in order and put back in the original array, giving us a sorted sequence. But, if ﻿NNN﻿ grows a lot in comparison to ﻿nnn﻿, then the performance deteriorates.


Properties:

  • Stable-sorting: bucket sort uses stable sorting, meaning that the order of duplicate keys stays the same, so if in the original array two entries with the same key are in some order, this order is retained after sorting.


Radix sort

Implementation:

Radix sort sorts a sequence of entries with keys that are pairs by applying stable bucket-sort twice: first using one component of the pair as the key and then using the second.


We should first sort using the second component actually and then the first, so it becomes lexicographically correct (so a sequence ﻿((1, 2), (4, 2))\left(\left(1,\ 2\right),\ \left(4,\ 2\right)\right)((1, 2), (4, 2))﻿ is first sorted by the second numbers (the 2's) and then the first). Though we can switch this order around, this can only work if we apply bucket sort recursively within each bucket formed in the previous iteration (MSD - Most Significant Digit). This could be faster than LSD (Least Significant Digit)


The time complexity of this is ﻿O(d(n+N))O\left(d\left(n+N\right)\right)O(d(n+N))﻿ as bucket sort is applied ﻿ddd﻿ (length of the to be sorted sequence or of a word, as seen in the slides) times.
