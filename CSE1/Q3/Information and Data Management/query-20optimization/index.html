Query optimization

  • The query optimization determines the specific order of the relational algebra operators, using an operator tree
  • Single relational algebra operators can be evaluated using one of several different algorithms
  • The evaluation plan is an annotated expression that specifies a detailed evaluation strategy / algorithms


Query optimizer

A query optimizer translates a naïve query plan (the query you get when you translate SQL into relational algebra) into an efficient plan by choosing which physical operators to use and in which sequence and groups they are executed


We make a distinction between heuristics (rules of thumb) and cost-based (educated guess) optimization. Most DBMS use a hybrid approach


Preparing the query

Basic mapping from query languages into a suitable internal format results in the canonical operator tree, but is very unoptimized


All evaluation plans are usually tree shaped sequences of the relational algebra operators (operator trees)

  • Relations as leaf nodes
  • Operators as internal nodes, with one or more children


Choosing the right plan

An exhaustive search strategy for finding the best plan for each query is prohibitively expensive.


Strategy using simple heuristics:

  • Apply selections as early as possible.
  • This keeps the number of intermediate results small, thus further operations will have to be applied to less records
  • As selections are commutative and associative, we can also break up conjunctive select statements, preparing for further optimization
  • We can also push selections as far down into the tree as possible
  • Apply projections as early as possible
  • Avoid Cartesian product, or if unavoidable, use them as late as possible
  • Use pipelining for adjacent unary operators


Algebraic query optimization

Relational algebra usually allows for alternative yet equivalent evaluation plans. All transformations are based on a set of relational algebra equivalences

  • Algebra allows for symbolic calculations
  • Transformation rules transform an operator tree into another, equivalent tree, step by step
  • Results of a query are never affected by transformations

Cost estimations

In IDM we cover the following cost estimations:

  • Simple estimations for result set sizes, result set sizes only rely on the relational algebra operator tree
  • Simple estimations for block accesses, block accesses rely only on the actual algorithm used for implementing algebra operators


Query execution

The naïve approach would be creating a temporary file for each sub result and using those as inputs for other operators. This is really time expensive


Implementing operators

Relation scan

Used to locate and retrieve records that fulfill a selection condition


Linear Search

Performs linear search over a relation, fetches pages on disk that contain records and scans those records.


The cost

  • In general: ﻿brb_rbr​﻿, so the blocks of the relation
  • ﻿≤br\le b_r≤br​﻿ if a condition refers to a primary attribute and we can stop early


Binary search

Performs binary search over an ordered relation. Thus we typically use this with a primary key.


This algorithm takes the median page and takes the next pages accordingly. The cost therefore is ﻿⌈log⁡2(br)⌉\lceil\log_2\left(b_r\right)\rceil⌈log2​(br​)⌉﻿



Index scan

Used to get the records from an indexed field.


The costs in general are the costs for finding the index + the costs for retrieving results. More specifically:

  • Tree-based index:
    • Costix=height of the tree\text{Cost}_{ix}=\text{height of the tree}Costix​=height of the tree (if the whole index is on disc)
    • Costix=height of the tree−1\text{Cost}_{ix}=\text{height of the tree} - 1Costix​=height of the tree−1 (if the root index of the tree is always in memory)
    • Costix=height of the tree−2\text{Cost}_{ix}=\text{height of the tree} - 2Costix​=height of the tree−2 (if the root and the first index layer are always in memory)
    • Costix=0\text{Cost}_{ix}=0Costix​=0 (if the whole index is in memory)
  • Hash index:   * Costix=0\text{Cost}_{ix}=0Costix​=0 (if the whole hash index is in memory)

Scan with conditions

For a conditional selection of the type ﻿R θ valueR\ \theta\ valueR θ value﻿ (﻿RRR﻿ is relation, ﻿θ\thetaθ﻿ a predicate), a linear scan / binary search can be used.


If a primary index is available

  • for ﻿θ∈{&gt;, ≥}\theta\in\left\{&gt;,\ \ge\right\}θ∈{>, ≥}﻿, find first value and scan the rest of the relation
  • for ﻿θ∈{&lt;, ≤}\theta\in\left\{&lt;,\ \le\right\}θ∈{<, ≤}﻿, do not use index, but scan relation until found


If a secondary index is available

  • for ﻿θ∈{&gt;, ≥}\theta\in\left\{&gt;,\ \ge\right\}θ∈{>, ≥}﻿, find first value and scan the rest of the index to find pointers to this record
  • for ﻿θ∈{&lt;, ≤}\theta\in\left\{&lt;,\ \le\right\}θ∈{<, ≤}﻿, scan index and collect record pointers until value is found


Complex conditions

Selection conditions can be joined with logical junctors: and, or and not


Depending on the existence of indexes, in some cases a combination of the above algorithms can be used.

  • For conjunctions, always choose the operation with the smallest selectivity for the relation scan and then test all other conditions in memory
  • For disjunctions, make sure that each block is only fetched once
  • For negations, usually a linear scan is needed


Access path

Predicate splitting

  • Deciding for a single-index access path is implemented by predicate splitting.
  • When multiple attributes are involved in a condition, just use one index for the most selective attribute, and scan for the other attribute’s conditions in the blocks retrieved by the index.
  • Predicate splitting leads to suboptimal, but sufficient results.


Multiple-index access paths

Deciding for multiple-index access path helps, if several selection conditions are indexed with secondary indexes

  • For each attribute condition, use an available index to retrieve the list of block IDs with potential results
  • Intersect the block ID lists
  • Scan for query results in the remaining blocks
  • This is typically more performant than predicate splitting


Selectivity reordering

Selectivity reordering uses commutativity to rearrange binary operations, as the order of operations doesn't matter in that case.


To make this more efficient, the most restrictive operations should be applied first.


Sorting operations

Sometimes an operator needs to sort records in a relation that is not already stored in sorted order. With a secondary index on the sort key the relation can be read in-order.


If the entire relation fits into memory, sorting is no problem. If only some pages of a relation can be fetched into the DB buffer, external sorting is needed. The most common way of doing this is merge sort.


Projection

Projection can be implemented by performing a projection on each tuple. This generally requires a complete relation scan, since relations are split horizontally over blocks (row stores).


In relational algebra, duplicates are removed if non-key attributes are involved. In SQL, this sometimes happens (but is very expensive)


Joins

  • Joins usually have to access two different relations. Only records having a counterpart in the second relation are in the result table
  • Size of join results can be estimated via the reduction factor of the join condition and the number and overlap of distinct values in both relations
  • Metrics are necessary for optimizing joins
  • Join Result Size
  • Join Cost
  • Easiest metric: size of intermediate results, the larger the intermediate results, the slower
  • Better metric: block accesses, block accesses are the biggest bottleneck, depends on the join implementation. Also consists of costs for writing the result


Database statistics

By weighting specific statistics, cost-optimizers make assumptions about the system’s bottleneck (e.g. I/O or CPU). For each node in the operator tree, the cost of performing and the size of the result of the corresponding operation must be estimated


The estimation of the costs starts with simple parameters (maintained in the system dialog)

  • Buffer size
  • Cardinality of the input base relations
  • Relation size
  • Available access paths


An important factor for finding good plans is to keep intermediate results of operators small. We can estimate the real result size by reduction factors, how to do so depends on the type of condition.
