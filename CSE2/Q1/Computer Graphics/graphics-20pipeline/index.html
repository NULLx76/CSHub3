Graphics pipeline

Camera movement and movement of the objects in an opposite direction is similar. So: we don't have to more the camera if we know how to move objects.


How do we move objects? The easiest way is just to add a transformation ﻿TTT﻿ vector to the coordinates. We can notate this by ﻿P′=P+TP&#x27;=P+TP′=P+T﻿. Notice that this transformation is not linear, as the zero vector will also move.

How do we rotate objects? Use a rotation matrix ﻿RRR﻿. We can notate this by ﻿P′=RPP&#x27;=RPP′=RP﻿


This is becoming quite complicated and hard to optimize in hardware. We want to have a notation which is simple, concise and allows for concatenation of operations. It would also be nice if we can allows projections and keep everything linear.


Homogeneous coordinates

We can do this though, using homogeneous coordinates and projective geometry. There we have a ﻿nnn﻿-dimension space which is represented by ﻿n+1n+1n+1﻿-dimensional coordinates without the zero vector.


We also get a new equivalence relation: two points ﻿ppp﻿ and ﻿qqq﻿ are equal iff there exists an ﻿a≠0a\ne0a​=0﻿ such that ﻿p×a=qp\times a=qp×a=q﻿. So ﻿(1,2,1)=(2,4,2)\left(1,2,1\right)=\left(2,4,2\right)(1,2,1)=(2,4,2)﻿.


But why do we use 3 coordinates for a 2D image?

Two points are the same if they lie on a line through the origin ﻿(0,0,0)\left(0,0,0\right)(0,0,0)﻿. So if points that are the same they are along a line, we lose a dimension (the line). This added axis is called ﻿WWW﻿ to underline the special role of the extra dimension.


You can make a connection between this 3D space and a normal 2D space. The traditional ("affine") 2D space can be embedded as a plane in the 3D space at ﻿W=1W=1W=1﻿ . As coordinates in the projective space on a line are the same, there is only one intersection with the 2D affine plane per coordinate.


How to go from projective to affine?

Divide by the last coordinate and keep the ﻿(x,y)\left(x,y\right)(x,y)﻿: ﻿(2x, 2y, 2)/2=(x, y, 1)⇒(x,y)\left(2x,\ 2y,\ 2\right)/2=\left(x,\ y,\ 1\right)\Rightarrow\left(x,y\right)(2x, 2y, 2)/2=(x, y, 1)⇒(x,y)﻿


How to go from affine to projective?

We can just map ﻿(x,y)⇒(x,y,1)\left(x,y\right)\Rightarrow\left(x,y,1\right)(x,y)⇒(x,y,1)﻿


But why?

When ﻿W=0W=0W=0﻿, we actually have a way to describe points at infinity. Normally, you wouldn't have the possibility to describe this normally.

Depth

As mentioned previously, we can still have the problem of two objects which are projected onto the same 2D coordinate. In order to decide which object to draw, we use the depth test. For each pixel, we keep track of the stored distance and we only update the pixel if the new distance is nearer.


But we don't actually need the "true depth", the real, actual value, we only need to be able to compare the depth values. Thus for this we can use a monotonic (growing) function.


If the object is too far or too close, we just don't draw it. This is called a clipping plane. This is illustrated here:

Image taken from slides by Elmar Eisemann


Note that if this clipping plane is very small, objects can appear out of nowhere (which is what happened in old games).

This matrix looks kind of similar to the projection matrix, with a few differences:

  • We divide by aspect, it doesn't make sense to cut out a square area for a ﻿16:916:916:9﻿ screen
  • Can be derived relatively easily if you try to map the region between near and far into values between ﻿−1-1−1﻿ and ﻿111﻿
  • The z-axis is inverted, this is how we usually do it for whatever reason


This matrix maps the perspective frustum into a unit cube:

Image taken from slides by Elmar Eisemann


Note that the ﻿zzz﻿ is not linear, so the middle of this cube is not at ﻿z=0z=0z=0﻿. This side-effect is actually useful, since the values are denser the closer they are to you, so the closer, the more precise.


If the triangle to be drawn does not entirely lie inside the cube, we need to cut it up. We do this by cutting them up and then re-triangulating them, so that we can still draw triangles.


In the end, we draw pixels only when the center of the pixel lies inside of the triangle


Conclusion

Summarized, in order to get our final result, we multiply a few matrices:

  1. Model matrix: as the model center is often not ﻿(0,0,0)\left(0,0,0\right)(0,0,0)﻿, we first need to transform our model into world space using the model matrix
  2. View matrix: as the camera will look at ﻿(0, 0, 1)\left(0,\ 0,\ 1\right)(0, 0, 1)﻿, we need to transform our world such that everything is in the right position in comparison to the camera, which we will do with the view matrix. This transforms everything into camera space
  3. Projection matrix: everything is now placed where we want it, but now we need to map everything onto the screen, depending on the depth value. This we do using the projection matrix, which maps everything into homogeneous coordinates.
