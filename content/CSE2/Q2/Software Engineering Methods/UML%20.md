<h1 id="uml">UML</h1><p>UML is a well-defined industry standard which tries to reduce risks by documenting assumptions. </p><p><br></p><p>There are three perspectives in drawing UML diagrams:</p><ul><li><strong>Conceptual</strong>: represent domain concepts (ignore software issues)</li><li><strong>Specification</strong>: focus on visible interfaces and behavior (ignore internal implementation)</li><li><strong>Implementation: </strong>document implementation choices (most common, but least useful)</li></ul><p><br></p><h2 id="use-case-diagram">Use case diagram</h2><p>A use case diagram documents the behavior from the user’s points of view.</p><p><br></p><p><strong>User</strong>: anything that interacts with the system. E.g. person, hardware device, etc.</p><p><br></p><p><strong>Notation</strong></p><ul><li>(denoted by a a stickman) An actor represents a user of the system, which can be anything external to the system which interacts with it!</li><li>A use case is represented as a named oval. It corresponds to a task which has to be done with support from the system under development.</li><li>A line connects an actor to a use case if the actor may interact with the system to play some part in the task</li></ul><p><br></p><p><strong>Actors</strong></p><p>We make a distinction between a <strong>primary </strong>- which acts on the system - and <strong>secondary </strong>- which is invoked/used by the system - actor.</p><p><br></p><p><strong>System boundary</strong></p><p>You can also draw a box around the use cases, a <strong>system boundary. </strong>This could be useful when there is a complex system which is split into different subsystems: one use case diagram per subsystem.</p><p><br></p><p><strong>Relationships</strong></p><p><img src="https://i.imgur.com/PEFQQEY.png" width="289"></p><p><em>Image taken from slides by Annibale Panichella</em></p><p><br></p><p>Here:</p><ul><li><strong>Includes </strong>indicate that a use case is included as parts of other use cases. This allows for factoring out common behavior from two or more original use cases. Or: a <strong>required </strong>part</li><li><strong>Extends </strong>indicate that a use case extends the behavior of other core use cases (the array goes from the "exceptional" case to normal case). Or: an <strong>optional </strong>part</li><li><strong>Generalizations </strong>allow for a relationship between a general use and a more specific use case</li></ul><p><br></p><p><strong>Examples</strong></p><p><img src="https://i.imgur.com/JREI73o.png" width="375"></p><p><img src="https://i.imgur.com/9NTK3rN.png" width="371"></p><p><img src="https://i.imgur.com/U8DAxUG.png" width="366"></p><p><em>Images taken from slides by Annibale Panichella</em></p><p><br></p><p><strong>Intent</strong></p><p>A use case diagram shows all use cases for a given system. Each use case is described in natural language. This natural language description describes the <strong>normal flow</strong> of a system (what's supposed to happen) and an <strong>alternative flow</strong> (what could also happen, e.g. authentication failure).</p><p><br></p><h2 id="sequence-diagrams">Sequence diagrams</h2><p>A sequence diagram provides a dynamic view of a use case scenario by showing the interactions among a set of objects in temporal order.</p><p><br></p><p><strong>Notation</strong></p><ul><li>Objects are denoted using vertical bars</li><li>Events or message dispatches are shown as horizontal arrows from the sender to the receiver</li><li>Conditions are denoted with rectangles labelled with "<strong>opt</strong>" and the guard condition is written in square brackets close to the label</li><li>If/then/else statements can be represented by using "<strong>alt</strong>" and writing <strong>[else]</strong></li><li>Loops can be represented by writing "<strong>loop</strong>" with the iteration clause nearby</li></ul><p><br></p><p><strong>Concurrency</strong></p><p>Concurrency can also be described in a sequence diagram:</p><p><img src="https://i.imgur.com/MK9VqnH.png" width="600"></p><p><em>Image taken from slides by Annibale Panichella</em></p><p><br></p><p><strong>Examples</strong></p><p><img src="https://i.imgur.com/wEhPtLK.png" width="414"></p><p><img src="https://i.imgur.com/mDGMWGP.png" width="427"></p><p><img src="https://i.imgur.com/KXnuj0s.png" width="414"></p><p><em>Images taken from slides by Annibale Panichella</em></p><p><br></p><h2 id="statechart-diagrams">Statechart diagrams</h2><p>Statechart Diagrams capture the behaviour of an individual class or object in response to interactions with other objects inside or outside the system.</p><p><br></p><p><strong>Notation</strong></p><p>The elements of statechart diagrams are as follows:</p><ul><li><strong>States </strong>are shown as a named box with rounded corners. Each class has one or more states</li><li>A <strong>transition </strong>between states is shown as arrows, this is caused by an event or a change in some guard condition</li><li><strong>Events </strong>cause transitions between events and are shown as messages on the transition arrow</li><li><strong>Start and stop markers </strong>are shown as black blobs</li><li><strong>Superstates </strong>are a set of states that have a common transition to a particular event and can thus be grouped into superstates</li></ul><p><br></p><p><strong>Transitions</strong></p><p>A <strong>transition</strong> is a response to an external event received by an object in a given state. They:</p><ul><li>may invoke an operation and cause a change in state</li><li>send an event to an external object</li><li>syntax is <strong>event(arguments) [condition]</strong></li></ul><p><br></p><ul><li>An <strong>operation </strong>is an atomic action invoked by a transition. An <strong>entry operation </strong>occurs whenever a transition is made into this state, and the <strong>exit operation </strong>is triggered when a transition is made out of this state</li><li>An <strong>activity </strong>is an ongoing operation that takes place while the object is in a given state. Internal transitions don't change state so entry and exit operations are not performed</li></ul><p><br></p><p><strong>Concurrency</strong></p><p>State diagrams can show forking and merging of concurrent submachines. We do this by drawing a nested state where the internal behavior of the state is made up of independently executing regions. Regions are separated by dashed lines:</p><p><img src="https://i.imgur.com/Y5dnAE3.png" width="448"></p><p><em>Image taken from slides by Annibale Panichella</em></p><p><br></p><ul><li>A labeled transition out of any of the substates terminates all of the substates</li><li>An unlabeled transition out of the overall state waits for all the substates to terminate</li></ul><p><br></p><p><strong>Composite statecharts</strong></p><p>A state which has substates (nested states) is called a composite state. A nested state machine should have at most one initial state and one final state.</p><p><br></p><p>Composite states may be depicted either as high-level or low-level views. “Stubbed transitions” indicate the presence of internal states: Internal and terminal substates are shown as black spots and “bulls-eyes”.</p><p><img src="https://i.imgur.com/h64fTcF.png" width="257"></p><p><em>Image taken from slides by Annibale Panichella</em></p><p><br></p><p><strong>Model driven testing</strong></p><p>You can use UML diagrams to do model-driven testing. You can aim for:</p><ul><li>All-stage coverage: each state should be covered at least once</li><li>All-transition coverage: each transition should be covered at least once</li><li>All-path coverage: each path should be covered at least once</li></ul><p><br></p><p><strong>Examples</strong></p><p><img src="https://i.imgur.com/t3vh8Qy.png" width="219"></p><p><img src="https://i.imgur.com/hPmz70o.png" width="428"></p><p><em>Images taken from slides by Annibale Panichella</em></p><p><br></p><h2 id="class-diagram">Class diagram</h2><h3 id="class">Class</h3><p>The Class diagram is a graphical notation used to design and visualize classes in object-oriented systems. A class diagram is a static structure diagram that describes:</p><div style="white-space: normal;" class="markdown-body"><ul>
<li>Classes in the system</li>
<li>Class attributes, with syntax <code>name: type = initialValue</code></li>
<li>Operations (methods), with syntax <code>name(parameter: type = defaultValue, ...): resultType</code></li>
<li>The relationships among objects</li>
</ul>
</div><p><br></p><div style="white-space: normal;" class="markdown-body"><p>Here,</p>
<ul>
<li><code>+</code> means public</li>
<li><code>#</code> means protected</li>
<li><code>-</code> means private</li>
<li>an underlined attribute is <code>static</code></li>
</ul>
</div><p><br></p><div style="white-space: normal;" class="markdown-body"><p>On top of the diagram you denote what kind of class it is, like <code>interface</code>, and the <code>{abstract}</code> is a user-defined property.</p>

Some contraints:</p><div style="white-space: normal;" class="markdown-body"><ul>
<li><code>{ordered}</code>, one class has an ordered collections of another class</li>
<li><code>{implicit}</code>, the relation is only conceptual</li>
<li><code>{changeable}</code>, links between objects may be added, removed and changed freely</li>
<li><code>{addonly}</code>, new links may be added from an object on the opposite end of the association</li>
<li><code>{frozen}</code>, a link may not be modified or deleted</li>
</ul>
</div><p><br></p><p><strong>Aggregation &amp; composition</strong></p><p>An aggregation shows that an object is part of an object of another class</p><p><br></p><p>A composition is a special kind of aggregation, which imposes further restrictions: if the whole object is copied or deleted, its parts are copied or deleted with it.</p><p><img src="https://i.imgur.com/DmK30tQ.png" width="437"></p><p><em>Images taken from slides by Annibale Panichella</em></p><p><br></p><p><strong>Association classes</strong></p><p>Sometimes the way in which two objects are associated is just as important as the objects themselves. The corresponding data (attributes) do not belong to any of the classes involved in the association.</p><p><br></p><p>We can then use an association class, which can be denoted as such:</p><p><img src="https://i.imgur.com/6Leqe0x.png" width="388"></p><p><em>Image taken from slides by Annibale Panichella</em></p><p><br></p><p><strong>Qualified associations</strong></p><p>A qualified association has a qualifier value that is used to select an object (or objects) from a larger set of related objects, based upon the qualifier key.</p><p><br></p><p><img src="https://i.imgur.com/6CPuL5K.png" width="600"></p><p><em>Image taken from slides by Annibale Panichella</em></p><p><br></p><p>Note that multiplicity is reduced to one, because it implies selection from a larger set.</p><p><br></p><p><strong>Generic classes</strong></p><p>Generic classes are depicted with their types shown in a dashed box, and you can bind the types from a specific subclass:</p><p><img src="https://i.imgur.com/JQLN9RW.png" width="544"></p><p><em>Image taken from slides by Annibale Panichella</em></p><p><br></p><p><strong>Utility classes</strong></p><p>A <strong>utility </strong>is a grouping of global attributes and operations. It is represented by the stereotype "&lt;&lt;utility&gt;&gt;". The attributes are already interpreted as static, thus no underline is required.</p><p><br></p><h2 id="component-diagram">Component diagram</h2><p>A component diagram depicts the relationship between different components in a system. In UML 2.x, a component is a logical (coherent) module of classes that represent an independent subsystem. In principle, a component can be replaced, reused or sold separately.</p><p><br></p><p><em><img src="https://i.imgur.com/MjgnXhB.png" width="0"></em><img src="https://i.imgur.com/MjgnXhB.png" width="0"><img src="https://i.imgur.com/MjgnXhB.png" width="413"></p><p><em>Image taken from slides by Annibale Panichella</em></p><p><br></p><h2 id="examples">Examples</h2><p><strong>Use case diagram</strong></p><p><img src="https://i.imgur.com/bfmDzYk.png" width="438"></p><p><br></p><p><strong>State diagram</strong></p><p><img src="https://i.imgur.com/mOn4Mst.png" width="403"></p><p><br></p><p><strong>Sequence diagram</strong></p><p><img src="https://i.imgur.com/EcD831x.png" width="294"></p><p><br></p><p><strong>Component diagram</strong></p><p><img src="https://i.imgur.com/A4mazeQ.png" width="356"></p><p><br></p><p><strong>Class diagram</strong></p><p><img src="https://i.imgur.com/PJvHD5r.png" width="381"></p>
