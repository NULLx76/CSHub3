Very basic core, extension standards for additional datatypes, numbers and timing.

Reasons for HDLs:

  • Specification
  • Simulation
  • Synthesis (automatic hardware generation)

For synthesis, not all VHDL constructs can be used as VHDL can not convert them to hardware properly.

concepts in VHDL

  • entities
  • ports (I/O)
  • architecture
  • connections (signals) which have values: time value pairs
  • event systems and delays for timing

Two types of VHDL designs:

  • functional/behavioral - functionality of a component. aAnd gates, or gates, xor gates, not gates, propagation delays.
  • structural - with components, and their connections. Connecting things

Datatypes in vhdl

  • bit (0 or 1)
  • bit_vector (a downto b) (difference between a and b is the bitwidth)
  • std_logic/std_logic_vector -> has more values than 0 and 1: weak, 1, 0, don't care, undefined, unset, high impedance, etc
entity xor_gate is 
	port ( 
		a, b: in bit; -- bit, so 1 or 0
		z: out bit 
	); 
end and_gate; 

architecture abc of xor_gate is -- link this architecture to this xor gate.
begin 
z <= a xor b after 10 ns; -- z is assigned to by a xor b, after 10 ns. If one of the inputs toggles on and off again in these 10 ns the output does not update.
end abc;

Order of text in architecture is not important.

Importing std_logic and std_ulogic:

library IEEE;
use IEEE.std_logic_1164.all;

Repeat this for every entity.

Intermediate signals:

architecture abc of some_circuit is
signal s1, s2: std_ulogic; -- intermediate signals
constant gate_delay: time := 5 ns; -- constants

-- mark xor gate for use in this architecture
component xor_gate is 
port ( a, b: in bit; z: out bit ); 
component


begin 
some_label: s1 <= a xor b after gate_delay; -- use of constants and (optional) labels


-- mandatory label, name of component, port map maps (component port =&gt; this entity port)
xor: xor_gate port map (a =&gt; s1, b =&gt; s2)
-- also allowed. port map in same order as components inputs
xor: xor_gate port map (s1, s2)



end abc;

Note: leaf nodes always need to be behavioral. In other words, you can structurally link components together but the components themselves need to be behavioral.




Testbench

Uses an FG (Function generator) to test a DUT (Design under test). A test bench does not have inputs/outputs. It generates a signal which tests the DUT.



Signal assignment statements


-- Priority top to bottom
signal_assignment: signal1 <= [signal2 | constant] (after time)? (when signal2_value else signal_assignment)? ";"

with select_value select 
signal_assignment: signal1 <= ("(" [signal2 | constant] (after time)? ")" when select_value)* ";"

Delay models

  • Inertial delay model: for gates, standard model (just using `after time`)
  • Transport delay model: for wires. Specify using an intermediate value and `a <= transport b after time`
  • Delta delay: automatic delay when no delays are specified. This is the smallest time step the simulator supports.


Processes

Starts the behavior with a sensitivity list: `process (sensitive signals) is`. Within a process you can use variables and if statements. Variables do not have any time component as compared to signals. Operations inside a process are sequential - top to bottom.Even signal assignments. Normal signal assignments are concurrent.
