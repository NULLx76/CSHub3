Human Computer Interaction

  • HCI is the study and the practice of usability: creating products that people want & find effective to use
  • HCI tries to provide an understanding of the computer and the person using it, to make the interaction between them more effective and enjoyable


What is everyone good at?

One of the challenges in HCI is that people and machines "think" and look at each other differently.


Humans are good at adapting to "hard and fuzzy" things

Computers are good at "simple and sharply defined things"


Notice the skills are complementary: let humans do what they do best and vice-versa.


Usability

A usable system includes some bottom-line benefits to businesses

  • Increases productivity
  • Decreased user training time and cost
  • Decreased user errors
  • Increased accuracy of data input and data interpretation
  • Decreased need for ongoing technical support


Usability is relevant to:

We are trying to achieve efficiency, efficacy and satisfaction, for the following types of use cases:

  • Regular ongoing use
  • Learning (new users)
  • Infrequent use (on each reuse)


Besides this, it should:

  • Be used by people with the widest range of capabilities
  • Minimize the risk and the undesirable consequences of use errors
  • Allow for easy maintenance


User experience: "person's perceptions and responses resulting from the use and or anticipated use of a product, system or service."


Getting usability right before implementing systems is much less expensive than supporting a bad user interface.


To achieve usability..

.. a product needs take into account and be tailored around a number of factors:

  • Cognitive, perceptual and motor capabilities and constrains of people in general
  • Special and unique characteristics of the intended user population in particular
  • Unique characteristics of the users' physical and social work environment
  • Unique characteristics and requirements of the users' tasks
  • Unique capabilities and constraints of the chosen software and/or hardware and the platform for the product.


PACT

We can use the PACT framework to help design a system from all of the important perspectives. It stands for people, activities, contexts and technologies.


It's a cyclical process: you have people that do activities in context and try to find a technology that meets these requirements. As a result of these technologies, it creates new opportunities to improve on these requirements.


People

When designing a system, every person is different and you need to take into account a lot of factors which may influence the experience a user has when using this system.


When working towards this, you do the following:

  • Identify your users
  • Identify relevant characteristics
  • Identify user classes
  • Draw out implications of user characteristics
  • Represent key users as personas


Identifying users

You can ask a few questions in order to help you identify the users of your system:

  • Position in organization?
  • Direct or indirect users?
  • Users of current system?
  • Identified from marketing surveys & focus groups
  • Can you meet them, observe them? Find out how they feel about their task, computers, the web, smartphones and anything else relevant
  • Can you read the marketing department's ideas or read transcripts of focus groups?


User Characteristics:

  • Age
  • Domain experience
  • Computer experience
  • Degree of pressure
  • Frequency of use
  • Freedom of choice in system use
  • Direct vs indirect use


Levels of expertise

Expert users with detailed knowledge will behave quite differently from people without this expertise. You need to take this onto account when designing a system: it should work for these experts, occasional users and novices.


Users may well be novices at one application but experts at another.


User classes

"A sub-set of the total population of end-users who are similar in terms of their system usage and relevant personal characteristics"


Different user classes may need different sets of functions, interface styles and help material.


To describe core user classes personas are used. Peronas are made-up users that define core user classes. You define a lot of attributes, such as gender, age and background. In the end you will look at the personas you have created and you will take a look at whether these personas are helped properly by your system.


Activities

Task analysis is the method of analyzing people's jobs:

  • What people do
  • What things they work with
  • What they must know


Task decomposition

Using task decomposition, we can:

  • Describe the actions people do
  • Structure them with task-subtask hierarchy
  • Describe ordering constrains of subtasks


This can be achieved using Hierarchical Task Analysis (HTA), which is a graphical illustration of a task structure:

  • Text and diagrams to show hierarchy
  • Plans to describe order


Generating a task hierarchy

This task hierarchy can be created using the following steps:

  1. Get a list of tasks - ask the user what they are doing
  2. Group tasks into high level tasks
  3. Aim for 4-8 tasks at top level
  4. Decompose lowest level tasks further
  5. Stopping rules - how do we know when to stop decomposing?


This is an iterative process: how can we improve the HTA we created?

Uses of task analysis

Requirements capture and systems design

  • Lifts focus from system to use
  • Suggests candidates for automation (not everything needs human intervention)
  • Uncovers users's conceptual model


Detailed interface design

Based on the task analysis, we can start designing our system:

  • Hierarchy suggests menu layout
  • Task frequency guides layout choices
  • Existing task sequences guide dialogue design


Do note that task analysis is never complete because you can never foresee every action that a user would try to accomplish.


Contexts

Activities always happen in a context, so we need to take this into account. The three useful types of distinguishable contexts are:

  • Physical environment
  • Social context
  • Organizational context


Scenario

A scenario is a brief story that describes how and why the persona would use the product to complete a specific task in a specific context.


Tasks and scenarios should not contain information about the user interface, so same task analysis can be used for different types of interfaces.


Technologies

Technologies and their interfaces have evolved over the last time.


We define a few terms:

  • Visibility ﻿→\to→﻿ what is seen of an object
  • Affordance ﻿→\to→﻿ what operations and manipulation can be done to a particular object
  • Perceived affordance ﻿→\to→﻿ what a person thinks can be done to the object. This is influenced by what is visible.


PACT use-case

The information gathered by the PACT framework allows us to:

  1. Define user requirements
  2. Design the user-interface
  3. Evaluatethe interface/system
  4. Design help/manuals


HCI tasks

In HCI, we have the following steps:

  • Requirement elicitation (interviews, ethnography etc.)
  • Requirement analysis (scenarios, task analysis etc.)
  • Prototyping (low vs high fidelity prototypes)
  • Participatory and incremental system design


Prototyping


Evaluation

How can the results be evaluated?

  • Expert evaluation (heuristic evaluation, cognitive walk-through)
  • User testing (formative and summative methods, e.g. focus groups)
  • Experimental design (e.g. user experiments, A/B testing)


Possible improvements

  • Accesibility (e.g. W3C standards, systems for visual and motor impairments)
  • Interactive, multi-modal systems (e.g. conversational agents, augmented reality)
